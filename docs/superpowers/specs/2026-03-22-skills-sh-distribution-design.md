# Skills.sh Distribution for Cogentic DS

**Date:** 2026-03-22
**Status:** Approved

## Summary

Publish the existing DS skill files to the skills.sh registry so consuming projects can install them with `npx skills add cogentic-co/ds`. This gives AI assistants (Claude Code, Cursor, Copilot, Cline, etc.) full knowledge of `@cogentic/ds` components, patterns, and design tokens when working in any project that uses the DS.

## Problem

The DS already has two skill files (`.claude/skills/ds-expert.md` and `.claude/skills/create-cogentic-app.md`) but they only work when developing the DS itself. Consuming projects don't benefit from them — AI assistants in those projects don't know which components exist, how to compose them, or which tokens to use.

## Approach

Restructure the existing skills into the skills.sh standard format (SKILL.md with YAML frontmatter) and place them in a top-level `skills/` directory for auto-discovery by the `npx skills add` CLI.

## Prerequisite: Public repo

`npx skills add` reads from a public GitHub repo. The current private repo needs to be published as a fresh public repo with no git history.

### Steps

1. **Create** new public repo `cogentic-co/ds` (or rename current to `-private` and claim the name)
2. **Copy** all source files — no `.git/` directory, no commit history
3. **Clean up** before the initial commit:
   - Remove Jira URL (`cogentic.atlassian.net/browse/SIG-490`) from CLAUDE.md and README
   - Replace `james@cogentic.co` with `user@example.com` in skill docs and preview components
   - Add `dist/` to `.gitignore` (build artifact, already generated by `prepublishOnly`)
4. **Publish** full source — `src/`, `app/`, `skills/`, tests, configs, everything
5. **Keep** the private repo as-is for reference/archive if desired

### Why full source public

- The DS is standard UI patterns with brand tokens — no competitive value in keeping source private
- Consumers can already read compiled source in `dist/` via the npm package
- Devs debugging props can click through to actual component code
- One repo, one truth — no syncing a public skills-only repo with a private source repo
- Shows engineering standards, testing, accessibility practices

## Design

### Directory structure

```
skills/
├── ds-expert/
│   └── SKILL.md          <- Main skill: how to use @cogentic/ds
└── create-cogentic-app/
    └── SKILL.md          <- Scaffolder skill
```

### SKILL.md format

Each file gets YAML frontmatter required by skills.sh:

```yaml
---
name: ds-expert
description: Expert guide for building UIs with the Cogentic Design System (@cogentic/ds). Covers all components, blocks, charts, workflow, hooks, design tokens, and patterns.
---
```

```yaml
---
name: create-cogentic-app
description: Scaffold new Next.js applications pre-configured with the Cogentic Design System (@cogentic/ds). Creates AppShell layout, routes, auth pages, and all configuration.
---
```

The body content of each skill is the existing content from `.claude/skills/ds-expert.md` and `.claude/skills/create-cogentic-app.md` respectively — no content changes needed.

### Consumer install

```bash
npx skills add cogentic-co/ds
```

This copies the skills into the consuming project's agent-specific skills directory (e.g., `.claude/skills/` for Claude Code). Works with 30+ agents.

### What the skill provides to consuming projects

- Full component catalog with props and signatures
- Correct import paths (barrel vs code-split entry points)
- Design token reference (semantic colors, fonts, radii, shadows)
- Composition patterns (AppShell, Card, Field, DataTable, etc.)
- Code examples for common pages (dashboard, settings, landing)
- Hook signatures and usage
- Block, chart, and workflow component docs
- Project scaffolding (create-cogentic-app)

### Changes to the DS repo

1. **Create** `skills/ds-expert/SKILL.md` — existing ds-expert.md content + frontmatter
2. **Create** `skills/create-cogentic-app/SKILL.md` — existing create-cogentic-app.md content + frontmatter
3. **Remove** `".claude/skills"` from `package.json` `files` array — skills.sh reads from the GitHub repo, not the npm tarball, so shipping skills in the package is unnecessary bloat
4. **Replace** `.claude/skills/*.md` with symlinks to `skills/*/SKILL.md` — avoids content duplication while keeping skills discoverable when working inside the DS repo
5. **Update** CLAUDE.md — add skill update step to New Component Checklist
6. **Update** CLAUDE.md — add `npx skills add` instructions to consuming section

### New Component Checklist addition

Add step 7 to the existing checklist in CLAUDE.md:

```markdown
### 7. Skill update
- [ ] Component added to `skills/ds-expert/SKILL.md` with signature, description, and category
- [ ] If new hook/block/chart: added to the appropriate section in the skill
```

This ensures the skill stays in sync as the DS evolves — same enforcement pattern as barrel exports, tests, previews, and metadata.

## What this does NOT include

- No CLI tool (unnecessary for a single-package DS)
- No project detection or `info --json` command (shadcn needs this because it's a copy-in system; we don't)
- No changes to the skill content itself — the existing ds-expert.md is already comprehensive
